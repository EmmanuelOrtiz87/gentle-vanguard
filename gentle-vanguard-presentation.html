VISION

¿Qué es Gentle-Vanguard?

Una plataforma agnóstica de ingeniería de software asistida por IA que orquesta agentes especializados, skills reutilizables y ciclos de calidad automatizados para transformar ideas en código validado.

01

Multi-Herramienta

Compatible con 10 herramientas AI. Se adapta automáticamente según el runtime detectado (opencode, claude-code, cursor, windsurf, cline, etc.)

02

Desarrollo Guiado por Specs

SDD Lifecycle: EXPLORE → PROPOSE → SPEC → DESIGN → APPLY → VERIFY. Cada fase ejecutada por el agente óptimo con el modelo adecuado.

03

Memoria y Mejora Continua

Engram memory system persiste decisiones, bugs y patrones entre sesiones. Post-session learning y self-healing automáticos.

METODOLOGÍA

SDD Lifecycle

Spec-Driven Development: cada feature nace de un spec, pasa por diseño, implementación y verificación

INIT → EXPLORE → PROPOSE → SPEC → TASKS → DESIGN → APPLY → VERIFY → ARCHIVE

🔍 Fase EXPLORE (BA)

Agente BA descubre necesidades, restricciones y criterios de éxito. Produce un documento de requerimientos estructurado.

🎨 Fase DESIGN (SAD)

Arquitecto de sistemas produce diseño técnico: componentes, API, base de datos, flujos.

💻 Fase APPLY (DEV)

Implementación con skills específicos por stack (React, Go, Django, Flutter, etc.)

✅ Fase VERIFY (QA)

Validación cross-workspace: usa playwright, pytest, testing-strategy y testing-coverage skills.

CALIDAD

Judgment Day

Protocolo de revisión dual adversarial. Dos jueces AI ciegos evalúan en paralelo, sintetizan, ofrecen fix cycle y re-evalúan.

Juez A Juez B → Síntesis → Fix Cycle → Re-Juicio → APPROVED → ESCALATED

⚠ Ejecución Automática

  • Pre-push hook: juzga cambios antes de push
  • Pre-merge-commit: juzga antes de merge PR
  • GitHub Actions: semanal (lunes 10:00 ART)
  • Evento session.ended también lo dispara

📜 Reglas Duras

  • Max 2 iteraciones de fix
  • Nunca declarar APPROVED si hay CRITICALs
  • Preservar posiciones minoritarias
  • Detectar sycophancy (cambio sin evidencia)

🎯 Gobernanza

  • Bloquea merge si judgment falla
  • Labels auto: approved/failed/pending
  • Coordina con CI pipeline
  • Reporta al event bus
PERSISTENCIA

Sesiones y Memoria

Cada interacción ocurre dentro de una sesión con estado. La memoria cruza sesiones.

🕑 Ciclo de Sesión

INACTIVE→ STARTING→ ACTIVE→ CLOSING→ CLOSED
  • Session ID único: session-YYYY-MM-DD-XX
  • Pre-process input para routing automático
  • Autostart valida git, engram, health checks
  • Session end: summary + memoria + auditoría

📖 Engram Memory System

  • Persiste decisiones, bugs, patrones, configs
  • Búsqueda semántica entre sesiones
  • Tipos: decision, architecture, bugfix, pattern
  • Post-session learning + gap analysis
  • Self-healing post-watchtower
  • NEXT_SESSION_GUIDE.md para continuidad
