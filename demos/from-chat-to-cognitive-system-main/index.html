01 · IA Generativa

Que es un LLM: prediccion, no pensamiento

Un Large Language Model es una red neuronal con billones de parametros entrenada para predecir el siguiente token. No "piensa": completa patrones estadisticos. La calidad del resultado depende de la calidad del contexto que le das.

02 · IA Generativa

Modelos que existen hoy

Gemini para razonamiento profundo, Opus para codigo limpio, Codex para testing, Copilot para autocompletado. La clave no es cual es "mejor" sino cual usar para cada tarea.

03 · IA Generativa

Open source chinos: frontier por centavos

Kimi K2.6, MiniMax M2.7 y GLM-5 ofrecen ~90% de la capacidad de Opus/GPT a 1/15 del precio. Frontier ya no es exclusivo de las grandes. El model router puede usar estos motores para tareas de alto volumen.

04 · IA Generativa

Que NO es un LLM

No es base de datos, no aprende de tus conversaciones, no entiende como humano, no es determinista. Si no sabes esto, vas a tener expectativas incorrectas y frustracion constante.

05 · Context Window

La pizarra de tamano fijo

El contexto es todo lo que el modelo "ve": system prompt, historial, archivos leidos, tu instruccion. La ventana de contexto es el tamano fisico de esa pizarra, medido en tokens.

06 · Context Window

Attention decay: mas no es mejor

A medida que llenas la ventana, el modelo no falla: degrada. Instruction drift, sesgo de recencia, codigo generico. Es sutil y peligroso porque el output "parece" correcto.

07 · Context Window

Compactacion: la amnesia forzada

Cuando la pizarra se llena, el sistema genera un resumen comprimido y borra los mensajes originales. Pierdes nombres de variables, decisiones de diseno, preferencias del usuario.

08 · Context Window

Como lo ataca la industria

Long context (1M+ tokens), RAG, fine-tuning, memory layers, vector stores. Cada approach tiene tradeoffs: costo, latencia, precision, complejidad. Nadie lo resolvio del todo.

09 · Context Window

Por que ninguna solucion es completa

Long context degrada calidad. RAG depende de embeddings. Fine-tuning es lento. Memory layers experimentales. Necesitas memoria que sobreviva sesiones, compactaciones y cambios de modelo. Ahi entra Engram.

10 · Chat vs Agente

Chat responde texto. Agente actua.

Un chat genera texto. Un agente lee tu codigo, edita archivos, ejecuta tests y guarda aprendizaje. Es la diferencia entre pedirle consejo a alguien y tener un par-programmer con acceso a tu proyecto.

11 · Chat vs Agente

Tools: las capacidades del agente

Terminal, editor de archivos, buscador, memoria. Cada tool es una capacidad externa. Sin tools es chat. Con tools es agente. Asi de simple.

12 · Evolución del Contexto

Todo empezo con AGENTS.md

La primera idea: un archivo markdown gigante con TODAS las reglas del proyecto. Estilo, patrones, anti-patrones, convenciones, workflows. El agente lo leia al inicio y "sabia" como trabajar.

13 · Evolución del Contexto

AGENTS.md crece infinito

Cada nueva convencion engordaba el archivo. En proyectos reales puede llegar a miles de lineas. Se come la ventana de contexto antes de que escribas tu primer prompt.

14 · Evolución del Contexto

Primera evolucion: Skills

En vez de un archivo gigante, un router (AGENTS.md liviano) que apunta a SKILL.md especificos. Solo se carga el contexto que necesitas para la tarea actual. Contexto dinamico bajo demanda.

15 · Evolución del Contexto

Segunda evolucion: sub-agentes

Skills resuelve QUE contexto cargar. Pero un solo agente sigue acumulando ruido. La solucion: delegar cada fase a un sub-agente efimero con contexto fresco. Nace, ejecuta, reporta y muere.

16 · Evolución del Contexto

El patron actual: todo junto

Orquestador (coordina) + Skills (contexto preciso) + Sub-agentes (ejecucion limpia) + Memoria persistente (continuidad). La arquitectura que resuelve las limitaciones del LLM con ingenieria.

17 · God Agent

Un agente que hace todo: por que degrada

"Analiza, disena, testea e implementa." Un solo agente llena su contexto al 80% antes de escribir la primera linea. Compila pero ignora las decisiones del paso 1.

18 · God Agent

Divide y venceras: orquestador + sub-agentes

El orquestador no trabaja: coordina. Delega cada tarea a un sub-agente especializado que recibe solo lo que necesita. Maxima precision, cero ruido acumulado.

19 · God Agent

Fresh context: arrancar limpio

Cada sub-agente nace con la pizarra vacia. Solo recibe las instrucciones de su fase. No arrastra ruido de fases anteriores. Esa es la ventaja arquitectonica clave.

20 · SDD Orchestrator

La filosofia: ingenieria aplicada a IA

En vez de "pedir y rezar", estructuramos el trabajo en una linea de montaje de expertos. Spec-Driven Development aplica principios de ingenieria de software al flujo con IA.

21 · SDD Orchestrator

Los roles especializados

Explorer (lee codigo), Proposer (define que), Spec Writer (requisitos), Designer (arquitectura), Task Planner (divide trabajo), Implementer (escribe codigo), Verifier (valida calidad).

22 · SDD Orchestrator

El DAG: flujo de dependencias

Proposal → [Specs || Design] → Tasks → Apply → Verify → Archive. Las fases se ejecutan en orden con checkpoints. Specs y Design pueden ir en paralelo.

23 · SDD Orchestrator

Ejemplo real: crear endpoint usuarios

/sdd:new → Explorer detecta stack → Proposer define alcance → Spec + Design en paralelo → Tasks divide en 4 → Implementer genera codigo → Verifier valida contra specs.

24 · SDD Orchestrator

Por que escala

Aislamiento de contexto, checkpoints de calidad, memoria institucional. El Implementer no se distrae con dudas del Explorer. Si el Verifier rechaza, no llega al PR.

25 · SDD Orchestrator

SDD Agent Team: la biblioteca

Open source. Orquestador delegate-only, sub-agentes con skills, DAG de dependencias, integracion con Engram. Listo para usar en cualquier stack.

26 · Engram

El problema: compactacion mata decisiones

Trabajas 2 horas definiendo arquitectura. El sistema compacta. El agente ya no recuerda que decidiste usar Clean Architecture con DI. Genera codigo generico.

27 · Engram

Como lo ataca la industria

RAG con vector stores, fine-tuning periodico, memory layers experimentales, long context windows. Cada uno tiene costo, complejidad y tradeoffs diferentes.

28 · Engram

Mi enfoque: memoria senal, no total

No guardamos todo (eso es ruido). Guardamos senales: decisiones, errores resueltos, patrones. Observaciones estructuradas con What/Why/Where/Learned.

29 · Engram

Progressive disclosure: 3 capas de eficiencia

En vez de volcar todo en el contexto, Engram revela informacion progresivamente. El agente decide que tan profundo ir segun lo que necesita. Minimo de tokens, maxima precision. 3 capas: mem_search (compacto: titulos + snippets), mem_context/timeline (contexto de sesion), y mem_get_observation (contenido completo sin truncar).

30 · Engram

Arquitectura: Go + SQLite + FTS5

Un binario. Sin dependencias externas. SQLite con Full-Text Search 5. Se conecta via MCP a 12 agentes: Claude Code, OpenCode, Kilo Code, Gemini CLI, Cursor, VS Code Copilot, Codex, Windsurf, Antigravity, Kimi Code, Qwen Code, Kiro IDE — y cualquier agente MCP-compatible.

31 · Engram

Las herramientas de memoria

Las 5 herramientas críticas. mem_current_project (nueva) confirma qué proyecto recibe tus saves antes de escribir. Y mem_save ahora captura solo el prompt asociado — no necesitás llamar a mem_save_prompt explícitamente.

32 · Engram

El Engram Loop: dos sesiones

Sesion 1: disenar → mem_save. Sesion 2: implementar → mem_search recupera decision → codigo consistente. Sin Engram la sesion 2 ignora la 1. Con Engram hay continuidad real.

33 · Engram

Dos formas de sincronizar: git o Cloud

Dos caminos válidos para sync, ambos sirven solo y para equipo. Git sync usa tu repo existente, sin infra extra, async (commit-based). Engram Cloud es un servidor self-hosted con dashboard web, autosync real-time, multi-user con roles. Elegís según tu necesidad: simplicidad o visibility.

34 · Engram

Cloud mode: el mismo binario engram, multi-máquina y real-time

Cloud no es un producto aparte — es un MODO del mismo binario engram. Pasás de SQLite local a Postgres + servidor con engram cloud serve. Sumás dashboard web (templ + htmx), autosync background con backoff exponencial, JWT auth + roles. 100% self-hosted — tu hardware, tu data, sin telemetría.

35 · Engram

Auto-detect: el proyecto lo decide tu cwd, no el LLM

Antes los agentes inventaban nombres distintos para el mismo proyecto y rompían la búsqueda. Ahora Engram detecta el proyecto desde tu cwd vía git remote (normalizado). Los write tools ya no aceptan project — el LLM no puede mentir. Si hay ambigüedad, te avisa con la lista de proyectos disponibles.

36 · Engram

Prompt capture automático: contexto que se guarda solo

mem_save ahora captura el prompt del usuario asociado, sin que tengas que pedirlo. SessionActivity recuerda el último prompt; mem_save lo dedupea y lo persiste con la observación. Resultado: cada decisión guardada viene con el contexto que la motivó. mem_save_prompt sigue disponible para casos donde un hook necesita anticiparse.

37 · Engram

Engram vs alternativas: por que lightweight gana

Sin servidor, sin embeddings, sin infra cloud. Un binario local que funciona offline. Para equipos que arrancan, la simplicidad es la feature mas importante.

38 · Skills Registry

El problema del prompt gigante

Si metes todas las reglas en el system prompt, saturas la ventana antes de empezar. El agente ignora reglas y genera codigo generico. Ya vimos por que en el modulo de evolucion.

39 · Skills Registry

AGENTS.md router + SKILL.md modulo

AGENTS.md como indice liviano que mapea intenciones a skills. Cada SKILL.md tiene patrones, anti-patrones y templates. Solo se carga lo que necesitas.

40 · Skills Registry

skill-creator: el equipo enseña al agente

El conocimiento institucional lo tiene el equipo. Con skill-creator cada persona crea sus propias skills: convenciones, patrones, checklists. El agente aprende del equipo real. Y la directiva contextual loading hace que el agente cargue automáticamente las skills relevantes al detectar triggers — no más "olvidé usar la skill X".

41 · Skills Registry

Contextual skill loading: orchestrator resuelve, sub-agents consumen

El orchestrator lee el skill-registry UNA vez al inicio de sesión y cachea las compact rules. Por cada delegación, matchea triggers (file context + task context) e inyecta las rules pre-digeridas en el prompt del sub-agent. Los sub-agents NO leen SKILL.md ni el registry — reciben todo pre-resuelto. Compaction-safe: si el caché se pierde, la siguiente delegación re-lee el registry.

42 · Stack y Bibliotecas

Ecosistema: Engram + SDD + Skills

Engram da continuidad, SDD da estructura, Skills da consistencia. Juntas resuelven los tres problemas del LLM: amnesia, degradacion y genericidad.

43 · Stack y Bibliotecas

Model router: un motor por fase

No existe el "mejor modelo". Gemini para arquitectura, Opus para implementacion, Codex para testing. La selección de modelo configurable por fase SDD ya está soportada por tres plataformas: OpenCode, Kilo Code y Kiro IDE (con subagents nativos vía model: frontmatter). Vos elegís qué motor maneja cada fase del DAG.

44 · Stack y Bibliotecas

Resultado: velocidad + consistencia + calidad

Velocidad por automatizacion. Consistencia por skills. Calidad por verificacion contra specs. El developer pasa de escribir codigo a dirigir y validar codigo.

45 · Stack y Bibliotecas

gentle-ai: un comando, cualquier agente, cualquier OS

Un comando instala todo el stack cognitivo. 12 agentes soportados, 3 sistemas operativos. El diagrama muestra el resto.

46 · Stack y Bibliotecas

Solo-agent vs Full delegation: dos formas de hacer SDD

Full delegation: cada fase SDD corre en un sub-agent con contexto fresco — el orchestrator coordina, los sub-agents ejecutan. Soportado por Claude Code, OpenCode, Kilo Code, Cursor, Gemini CLI, VS Code Copilot, Kimi Code, Qwen Code, Kiro IDE. Solo-agent: todas las fases corren inline en la misma conversación — el orchestrator ES el ejecutor. Engram da la persistencia entre fases. Soportado por Codex, Windsurf, Antigravity.

47 · Cierre

La diferencia: arquitectura alrededor del modelo

El modelo es el motor. Pero un motor sin chasis, sin direccion y sin frenos no es un auto: es un peligro. La arquitectura alrededor del LLM define si tenes un juguete o una herramienta.
