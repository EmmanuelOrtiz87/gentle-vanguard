VISION

¿Qué es Gentle-Vanguard?

Una plataforma agnóstica de ingeniería de software asistida por IA que orquesta agentes especializados, skills reutilizables y ciclos de calidad automatizados para transformar ideas en código validado.

Para gerencia: Es el "sistema operativo" de nuestra ingeniería con IA. Reduce ciclos, elimina riesgo de lock-in con herramientas, y construye memoria institucional automatizada.
01

Multi-Herramienta

Compatible con 10 herramientas AI. Se adapta automáticamente según el runtime detectado: opencode, claude-code, cursor, windsurf, cline, continue.dev, copilot, codex, antigravity, claude-generic.

02

Desarrollo Guiado por Specs

SDD Lifecycle: EXPLORE → PROPOSE → SPEC → TASKS → DESIGN → APPLY → VERIFY → ARCHIVE. Cada fase ejecutada por el agente óptimo con el modelo adecuado.

03

Memoria y Mejora Continua

Engram memory system persiste decisiones, bugs y patrones entre sesiones. Post-session learning, gap analysis y self-healing automáticos. 978+ observaciones almacenadas.

FUNDAMENTOS

Principios de Diseño

7 principios que gobiernan cada decisión arquitectónica

P1 Tool-Agnostic

Sin dependencia de una herramienta AI específica. Migrar de OpenCode a Claude Code o Cursor no requiere cambiar la plataforma.

P2 Spec-Driven

Cada feature nace de un spec validado. Sin spec aprobado, no hay implementación. SDD Gate bloquea PRs sin spec.

P3 Dual Review

Judgment Day: dos jueces AI ciegos revisan en paralelo. Síntesis, fix cycle, re-evaluación. Sycophancy detection.

P4 Memoria Persistente

Toda decisión, bug y patrón se persiste en Engram. La memoria cruza sesiones. Sin pérdida de contexto.

P5 Autonomía Progresiva

Auto-healing, auto-start, auto-learning. El sistema se recupera solo, inicia solo, y aprende solo de cada sesión.

P6 Calidad Blindada

28 quality gates, 16 workflows CI/CD, pre-commit hooks, pre-merge hooks. Nada pasa sin validación multi-capa.

RESILIENCE

Tolerancia a Fallos

Arquitectura de 3 niveles con failover automático

T1

Primary Execution

Activo Health check cada 5s, failover timeout 5s

  • Ejecución principal de comandos
  • Sincronización de estado cada 5s
  • Auto-recuperación en fallo
T2

Secondary Failover

Standby Health check cada 10s

  • Asume control si T1 falla >5s
  • Estado replicado desde T1
  • Transición transparente
T3

Tertiary Recovery

Recovery Health check cada 30s

  • Último recurso de recuperación
  • Timeout de failover: 60s
  • Modo degradado con funcionalidad mínima

📈 Métricas de Resilience

  • Uptime: Monitoreo continuo de todos los tiers
  • Failover Count: Número de conmutaciones registradas
  • Recovery Time: Tiempo medio de recuperación
  • Data Loss Events: Eventos con pérdida de datos (objetivo: 0)

💡 Auto-Recovery

  • Watchtower detecta anomalías y dispara recovery
  • Session manager repara sesiones huérfanas
  • Self-diagnosis: 90/100 health score actual
  • Karpathy baseline: 119 items de referencia

METODOLOGIA

SDD Lifecycle

Spec-Driven Development: cada feature nace de un spec, pasa por diseño, implementación y verificación

INIT→ EXPLORE→ PROPOSE→ SPEC→ TASKS→ DESIGN→ APPLY→ VERIFY→ ARCHIVE
Fase EXPLORE BA

El agente BA descubre necesidades, restricciones y criterios de éxito. Produce un documento de requerimientos estructurado con:

  • User stories y criterios de aceptación
  • Restricciones técnicas y de negocio
  • Análisis de impacto en el sistema existente
  • Recomendación de skill a utilizar
  • Tiempo estimado: 1-3 turnos de conversación
Fase DESIGN SAD

El arquitecto de sistemas produce el diseño técnico detallado:

  • Diagrama de componentes y relaciones
  • API contracts (request/response, endpoints)
  • Modelo de datos y esquema de base de datos
  • Flujos de proceso y state machines
  • Consideraciones de seguridad y escalabilidad
Fase APPLY DEV

Implementación con skills específicos por stack. El agente DEV:

  • Carga el skill correspondiente al stack (React, Go, Django, etc.)
  • Implementa siguiendo el spec y design aprobados
  • Escribe tests unitarios y de integración
  • Respeta convenciones del proyecto (modification protocol)
  • Workload guard si supera 400 líneas recomienda PR encadenado
Fase VERIFY QA

Validación cross-workspace completa:

  • Ejecuta tests existentes para detectar regresiones
  • Valida contra los criterios de aceptación del spec
  • Cross-workspace sync si aplica
  • Reporte de cobertura y calidad
  • Si pasa, avanza a ARCHIVE. Si falla, retorna a APPLY con hallazgos
METODOLOGIA

SDD Preflight & Gates

Mecanismos de control que garantizan la calidad del ciclo SDD

✈ SDD Preflight

Ejecutado antes del primer flujo SDD en una sesión. Establece:

  • Modo: interactivo (con usuario) o auto (autónomo)
  • Artifact store: openspec, engram, o ambos
  • PR strategy: PR único, encadenado, o directo a develop
  • Review budget: máximo de iteraciones de revisión

🔒 SDD Gate

Bloquea PRs en protected branches si:

  • No hay spec aprobado asociado al PR
  • El spec no tiene design correspondiente
  • Hay cambios sin tareas SDD asociadas
  • Workflow: sdd-gate.yml en GitHub Actions

📈 Workload Guard

Antes de implementación multi-archivo, verifica:

  • Si cambios estimados >400 líneas recomienda PRs encadenados
  • Si hay >5 archivos afectados sugiere separar por capa
  • Previene PRs masivos difíciles de revisar

🔄 Review Workload Guard

Script: scripts/utilities/review-workload-guard.ps1

  • Cuenta líneas estimadas de cambio
  • Si supera 400: recomienda chained PRs
  • Integrado en el flujo de delegación
  • Obligatorio antes de cualquier multi-file implementation
CALIDAD

Judgment Day

Protocolo de revisión dual adversarial. Dos jueces AI ciegos evalúan en paralelo, sintetizan, ofrecen fix cycle y re-evalúan.

Juez A Juez B → Síntesis → Fix Cycle → Re-Juicio → APPROVED → ESCALATED

⚠ Ejecución
  • Pre-push hook: juzga cambios antes de push
  • Pre-merge-commit: juzga antes de merge PR
  • GitHub Actions: semanal (lun 10:00 ART)
  • Evento session.ended: dispara juicio de la sesión

📜 Reglas Duras
  • Max 2 iteraciones de fix por ciclo
  • Nunca declarar APPROVED si hay hallazgos CRITICAL
  • Preservar posiciones minoritarias en el reporte
  • Detectar sycophancy (cambio de opinión sin evidencia nueva)
  • Evidencia debe ser citable y verificable

🎯 Gobernanza
  • Bloquea merge si judgment falla (CI gate)
  • Labels automáticos: approved / failed / pending
  • Coordina con pipeline CI/CD general
  • Reporta al event bus del sistema
  • Skill dedicado: skills/judgment-day/
Para equipo técnico: Judgment Day usa dos instancias del modelo GOV con temperatura 0.1. Cada juez produce un reporte estructurado (CRITICAL, WARNING, INFO). La síntesis combina ambos y detecta discrepancias. Fix cycle permite correcciones antes del veredicto final.

REFERENCIA

Glosario de Términos

Vocabulario técnico y de negocio usado en la plataforma

A

  • Agent: Cliente AI especializado con rol, temperatura y modelo asignado
  • Auto-delegation: Routing automático de intención → agente → skill
  • Adaptive profile: Perfil de comportamiento que se ajusta según la herramienta detectada

C

  • CodeGraph: Índice de código para navegación y análisis de impacto
  • Cross-workspace: Validación que abarca múltiples repositorios

E

  • Engram: Sistema de memoria persistente con búsqueda semántica
  • Executive Dashboard: Panel con métricas de negocio y estado del sistema

J

  • Judgment Day: Protocolo de revisión dual adversarial con fix cycle
  • Juez: Instancia de agente GOV que produce un veredicto estructurado

K

  • Karpathy Baseline: Conjunto de 119 items de referencia para detectar regresiones

M

  • MCP: Model Context Protocol — puente de comunicación entre herramientas
  • mem_save: Comando para persistir observaciones a Engram

Q

  • Quality Gate: Punto de validación obligatorio en el pipeline CI/CD

S

  • SDD: Spec-Driven Development — ciclo de vida de desarrollo guiado por especificaciones
  • Skill: Paquete reutilizable de instrucciones para un dominio específico
  • Sycophancy: Cambio de opinión sin evidencia nueva (detectado por Judgment Day)
  • Session: Unidad de interacción con estado, identificador único y ciclo de vida

T

  • Tier: Nivel de resiliencia (primary, secondary, tertiary)
  • Token Budget: Límite diario de consumo de tokens (120,000)
