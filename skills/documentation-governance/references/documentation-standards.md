# Documentation Standards

## 1. README Template

Use this order unless the project clearly needs something different:

1. Project name and one-line purpose.
2. Start Here.
3. Requirements.
4. Configuration.
5. Run / Start.
6. What It Shows or What It Does.
7. Architecture.
8. Local Automation.
9. Files to Know.
10. Prerequisites.
11. Notes.

## 2. Installation or Setup Guide

Use this order for onboarding docs:

1. Start Here.
2. Recommended Order.
3. Tool-by-tool installation steps.
4. Project repository setup.
5. Optional local memory or secrets steps.
6. Validation commands.

## 3. Technical Document

Use this order for technical overviews:

1. Summary.
2. Functional Goal.
3. Architecture.
4. Data Flow.
5. Main Views or Main Components.
6. Performance Handling.
7. Security.
8. Validation.
9. Maintenance Notes.

## 4. Code Review Template

Use this order for review notes:

1. Title, date, reviewer, scope.
2. Executive Summary.
3. Critical Issues.
4. Minor Issues.
5. Recommendations.
6. Quality Metrics.
7. Standards Checklist.
8. Conclusion.

## 5. Script Commenting Rules

1. Comment why something exists, not what the code already makes obvious.
2. Comment any fragile step, boundary condition, or cleanup requirement.
3. Prefer short comments before complex blocks.
4. Keep comments in English.
5. Do not let comments drift away from the actual script behavior.

## 6. Cleanup Checklist

Before closing a documentation task, verify:

1. The main entry point is obvious.
2. File and script names match the docs.
3. Old references are removed.
4. Temporary or legacy docs are not left in the active flow.
5. The language is English unless the project explicitly requires otherwise.
